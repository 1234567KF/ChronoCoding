## 二、按 Token 类型分拆

### 输入 Token 优化

| 手段         | 原理                                                                          | 技能                 | 节省率         |
| ------------ | ----------------------------------------------------------------------------- | -------------------- | -------------- |
| CLI 输出压缩 | Shell Hook 拦截 CLI 命令输出，90+ 压缩模式，去除框架冗余/运行时噪音/路径前缀  | lean-ctx             | 80-99%         |
| 文件读取压缩 | ctx_read 替代 Read，10 种读取模式（aggressive/entropy/task 等），重读 ~13 tok | lean-ctx             | 67-99%         |
| 依赖图谱     | 只读取变更文件及其依赖树，而非全量扫描                                        | kf-code-review-graph | 30-70%         |
| 记忆复用     | 跨会话 SQLite + 向量检索，避免重复读取文档/配置                               | claude-mem           | 100-300 tok/次 |
| 上下文压缩   | 90+ 压缩模式自动选择，智能识别已紧凑内容（如 `git diff --stat`）不压缩      | lean-ctx             | —             |

### 输出 Token 优化

| 手段       | 原理                                                                  | 技能               | 节省率 |
| ---------- | --------------------------------------------------------------------- | ------------------ | ------ |
| 原子协议   | 340+ 预定义 Lambda 原子替代自然语言指令（`!ta ct` vs "请执行任务"） | lambda-lang        | 66.5%  |
| 结构化提取 | LLM 驱动非结构化→JSON/CSV/YAML，减少冗余描述                         | kf-langextract     | ~20%   |
| 回调机制   | Agent 完成后主动通知，替代轮询 "完成了没" 的重复查询                  | claude-code-pro    | 80-97% |
| 文档一致性 | 减少因文档不一致导致的额外解释和修正                                  | kf-doc-consistency | 10-20% |

### 缓存命中优化

| 手段       | 原理                                              | 技能            | 节省率                    |
| ---------- | ------------------------------------------------- | --------------- | ------------------------- |
| 会话连续性 | context-mode 保持会话不中断，prompt cache 不断    | context-mode    | 缓存读 ≈ 输入 1/120 价格 |
| 智能跳过   | <3 文件不 spawn Agent，保留在主会话中（缓存不断） | claude-code-pro | cache 不中断              |
| 批量处理   | 合并多个小任务在一次调用中，共享 prompt cache     | 用户习惯        | 减少 cache miss           |
| **多Agent共享** | **同会话多Agent共享prompt前缀缓存,N agent≈1+(N-1)×2%成本** | **会话管理** | **N×→1.08×(N=5时)** |

## 三、按价格模型拆分（2026/05 定价）

> 以 deepseek-v4-pro 为例：输入 ¥3/MTok | 输出 ¥6/MTok | Cache 读 ¥0.025/MTok

| 优化手段           | 影响 Token 类型 | 价格影响                   | 说明                              |
| ------------------ | --------------- | -------------------------- | --------------------------------- |
| lean-ctx CLI 压缩  | 输入            | **省 80-99%**        | 直接减少每次 CLI 输出的输入 token |
| model-router Flash | 输入+输出       | **省 ~67%**          | Flash 价格约为 Pro 1/3            |
| ccp 跳过 spawn     | 输入+输出       | **省 15K tok/spawn** | 每次跳过的总分成本                |
| lambda-lang        | 输出            | **省 66.5%**         | 减少 Agent 间通信输出             |
| Cache 命中         | 输入            | **省 99.2%**         | ¥0.025 vs ¥3.0/MTok             |
| 轮询→回调         | 输入+输出       | **省 80-97%**        | 减少无效轮询的往返                |

## 五、控制变量测试要点

| 变量           | 说明                                                | 测试方法                       |
| -------------- | --------------------------------------------------- | ------------------------------ |
| 第1次 vs 第2次 | 第2次可能命中 prompt cache（输入价格差 120x）       | 同任务隔天/隔 session 复测     |
| 输入 vs 输出   | 输入 ¥3/MTok, 输出 ¥6/MTok, cache 读 ¥0.025/MTok | 分别记录三类 token             |
| 模型差异       | Pro 推理强但贵，Flash 速度快且便宜                  | 同任务两模型对比，LLM 评判质量 |
| 文件数量       | 1-2 文件应跳 spawn，10+ 文件必须 spawn              | 按文件数梯度测试               |
| 环境一致性     | 同机器、同时段、同网络                              | Docker 隔离或固定测试机        |
