## 蓝队方案 — 快省优先

### 总体立场

**快是杠杆，省是底线。** 任何加环节的优化，必须证明节省的时间 > 新增的时间，否则就是倒退。我们的核心原则：**能不等的就不等，能一次过的就不循环，能自动化的就不让人盯。**

---

### 1. Plan 审批关卡

**立场：反对阻塞式审批，支持查阅式输出。**

#### 为什么增加审批关卡会降低效率？

- **上下文切换成本高**：人类收到审批通知时需要中断当前工作，理解 Agent 的 Plan 上下文，然后决策。这个"进入状态"的成本往往比审批本身高一个数量级。对于每天 10+ 次任务，每次 interruption 成本约 15-20 分钟，积累下来反而拖慢整体交付。
- **瓶颈转移**：当一个人 + N 个 Agent 并行工作时，人类是唯一的单线程瓶颈。Agent 等人类审批时的空闲时间 = 资源浪费。ListenHub 的"1 人干 20 人活"场景下，人在 Plan 审批上堵住了，后面的 Agent 全部空转。
- **80/20 法则**：统计上，80% 的任务 Plan 是明确且正确的（因为 kf-alignment 已做了需求对齐），审批变成 rubber-stamping，徒增延迟。只有 20% 的新颖/高风险任务需要人类把关。
- **信任赤字**：如果 Agent 每次都需要人类确认才敢执行，说明 Agent 的能力不足——应该提升 Agent 的质量，而不是加审批来兜底。

#### 什么情况下审批是必要的？什么情况下是浪费？

| 场景 | 审批必要性 | 说明 |
|------|-----------|------|
| 数据库迁移 / 高危操作 | **必要** | 改生产数据、删表、改权限等不可逆操作 |
| 对外发布 / 上线 | **必要** | 涉及用户可见变更 |
| 简单 CRUD 功能 | **浪费** | Plan 几乎可预测，审批是走形式 |
| Bug 修复 | **浪费** | 修复方案通常明确 |
| 新功能开发（高复杂度） | **可能必要** | 取决于人类对方向的掌控需求 |
| 重构（大面积） | **有条件必要** | 只在影响核心架构时需要 |

#### 轻量替代方案：查阅式 Plan 输出

```
┌─ User: "/夯 实现登录功能"
│
├─ Step 1: Agent 生成 Plan + 自动执行
│   ├─ 输出: plan.md（自动写入，不阻塞）
│   └─ Agent 按 Plan 开始执行
│
├─ Step 2: 人类可随时查阅 plan.md
│   ├─ 发现方向偏差 → 中断 Agent，修正后重来
│   └─ 无偏差 → 继续执行（无需回应）
│
└─ Step 3: 最终交付时汇总 Plan vs 实际
```

**关键差异**：Plan 不是检查点（checkpoint），而是仪表盘（dashboard）。人类**可以看**但不**必须看**。这保留了人类对 scope 的掌控感，但不会让人类成为流程瓶颈。

**实现方式**：在 kf-spec / `/夯` 的输出中自动写入 `.claude/plans/latest-plan.md`，然后只需要一行提示：

> "Plan 已生成：可执行 `.claude/plans/latest-plan.md` 查阅。如需修改请在 30 秒内打断，否则自动继续。"

### 2. Review-Fix 反馈环

**立场：反对一刀切多轮循环，支持"一次 review + 条件触发重审"。**

#### 一次 review + AI 自动修复是否足够？

- **足够**，对于 > 80% 的代码。LLM 的 review 能力已足够发现绝大多数问题（风格、安全、常见反模式），且一次修复的准确率很高。
- **关键发现**：多轮 review 的边际收益递减极快。第一轮可发现 95% 的问题，第二轮能发现剩余 5% 中的 60%，第三轮之后基本是噪声。收益曲线是 logarithmic，成本曲线是 linear。
- **例子**：如果一次 review + fix 需要 5K tokens，第二轮需要重新 review 修改后的代码（5K）+ 可能的小 fix（1K）= 6K tokens。第三轮同理。三轮下来 token 消耗是单轮的 3x+，但质量提升可能只有 2-3%。

#### 多轮循环的 Token 和时间成本

| 轮次 | 成本构成 | 累计 Tokens（估） | 质量提升 |
|------|---------|------------------|---------|
| 1 | review(5K) + fix(3K) | 8K | 基线 |
| 2 | re-review(5K) + fix(1K) | 14K (+75%) | +2~3% |
| 3 | re-review(5K) + fix(0.5K) | 19.5K (+39%) | +0.5~1% |
| 4+ | 边际成本 > 边际收益 | — | < 0.5% |

**结论**：强制多轮循环是慢且贵的。需要的是"最差情况有兜底，一般情况不绕路"。

#### 什么场景下才需要多轮？如何做条件触发？

只有以下场景需要触发多轮 review：

1. **安全关键代码**（认证、加密、支付、权限）→ 至少 2 轮
2. **review 发现 P0 缺陷**（逻辑错误、数据竞争、内存泄漏）→ 自动修复后重审 1 轮
3. **跨文件重构**（涉及 5+ 文件变更）→ 重审变更文件
4. **人类主动要求** → 用户说"再 review 一遍"

**其余场景一律一次过**。风格问题、命名建议、小的返回值检查等，不需要第二轮。

#### 实现方案

```diff
- Stage 4: kf-code-review-graph → 出报告 → 结束
+ Stage 4: kf-code-review-graph → 出报告 → 按严重级别分类
+   ├─ P0/P1 缺陷 → 自动修复 → re-review（最多 1 轮重审）
+   └─ P2/P3 建议 → 直接修复 or 记录到 TODO（不重审）
+   └─ 安全/高敏模块 → 额外 1 轮重审
```

### 3. Spec 结构化增强

**立场：适度增强，够用就好。拒绝过度工程。**

#### 值得加的字段（高 ROI）

| 字段 | 原因 | 预期工作量 |
|------|------|-----------|
| **验收条件 (AC)** | 最核心的增强。减少"做完了但不对"的返工，AC = Agent 的终止条件，没有 AC 的 Spec 容易让 Agent 做过头或做漏。 | 小（模板字段 + 示例） |
| **边界情况 (Edge Cases)** | 高 ROI。Agent 默认倾向 happy path，显式列出边界能显著减少 bug。而且人类写边界情况的成本远低于 debug 成本。 | 中（需要人类思考，但 Agent 可辅助生成） |
| **依赖关系 (Dependencies)** | 有用但应自动化。可以从代码库/package.json 自动分析，不应由人类手动写。 | 小（自动检测 + 输出到 Spec） |

#### 加了反而负担的字段（负 ROI）

| 字段 | 原因 |
|------|------|
| **测试策略 (Test Strategy)** | 测试策略应由 Agent 根据代码自动生成，不需要写到 Spec 里。写测试策略 ≈ 写了两遍需求。让 Agent 自己决定如何测试更高效。 |
| **API 签名/数据模型** | 过深的细节反而限制 Agent 的灵活性。Spec 写"用户登录需要手机号+验证码"就够了，不需要提前定死 API 路径和参数名——让 Agent 在编码时根据框架惯例决定。 |
| **UI 像素级描述** | 除非是设计稿驱动的项目，否则在 Spec 里写 UI 细节是浪费。用 kf-ui-prototype-generator 出原型，Spec 只写功能逻辑。 |
| **数据库 Schema 详细设计** | Spec 阶段写表结构容易出现"需求还没定死就在设计表"的情况，导致后续改 Schema 成本高。应该在 Plan 阶段由 Agent 生成。 |

#### "够用就好"的 Spec 模板

```markdown
# Spec: {功能名称}

## 概述
{一句话说明此功能，不超过 50 字}

## 验收条件
- [ ] 条件 1：...（可测试的 pass/fail 标准）
- [ ] 条件 2：...

## 边界情况
- 异常场景 1：...
- 异常场景 2：...

## 依赖
- 前置依赖：{自动检测填充}
- 外部依赖：{自动检测填充}

## 备注
{任何需要人类备注的上下文}
```

**关键原则**：Spec 够细让 Agent 方向不偏，但留足空间让 Agent 做实现决策。Spec 是"边界"不是"牢笼"。

### 4. 测试前移

**立场：单元测试随代码走（赞同），集成/E2E 前移是陷阱（反对一刀切）。**

#### 单元测试应随代码生成（赞同）

- **理由**：单元测试验证独立模块的正确性，不依赖外部环境。随代码生成的边际成本极低（LLM 可以直接生成配套测试），但收益高——每行代码写出来就带着测试，相当于"测试驱动开发"的轻量版。
- **实现**：kf-spec 输出后，Agent 编码时自动生成对应的 `.test.ts` 文件，随代码一起提交。这本质上是"省"——等代码写完了再加单元测试，需要额外的上下文加载成本（重新理解代码、搭建环境）。

#### 集成/E2E 前移的陷阱

- **测试脆弱性**：集成测试依赖外部服务（数据库、API、第三方），在 Stage 1 的代码还远未稳定时写集成测试，会导致测试在迭代中被频繁破坏，维护成本飙升。
- **假阴性成本**：前移的集成测试很可能因为代码还没写完或者接口还没确定而失败。每次失败的告警都是噪声，噪声多了团队就不看测试了——这是最坏的结果。
- **前移 ≠ 质量更好**：测试的质量取决于代码的稳定性。Stage 1 的代码改 3 轮之后，Stage 1 写的 E2E 测试基本要重写，等于白做。

#### 区分"关键路径测试前移"和"全量测试前移"

| 类型 | 前移策略 | 理由 |
|------|---------|------|
| 单元测试 | **全量前移** | 低成本、高收益。Agent 写代码时顺手生成，不额外花时间 |
| 关键路径集成测试 | **有条件前移** | 只对核心链路（如登录→支付）写基础集成测试，不追求覆盖，只做冒烟验证 |
| 全量集成测试 | **不应前移** | 等代码稳定（Stage 2 后）再写，避免测试脆弱性 |
| E2E 测试 | **不应前移** | 等所有模块开发完成（Stage 3）再写，保持 browser-ops 现有的 Stage 3 定位 |
| 回归测试 | **不应前移** | 回归测试天然在最后，不应该打乱顺序 |

#### 建议的测试流水线

```
Stage 1（编码）: 单元测试随代码生成 ← 全量前移 ✓
Stage 2（集成）: 仅关键路径集成测试，剩余集成测试等代码稳定后补充
Stage 3（E2E）: browser-ops E2E 测试 ← 保持不变
Stage 4（回归）: 全量回归测试 ← 保持不变
```

### 5. 效率保障

优化不能拖慢现有流程。以下是四条铁律：

#### 铁律 1：所有优化必须有 opt-out

每一项新增环节（Plan 查阅、review 重审、测试生成）默认开启，但允许用户一键跳过：

```bash
/夯 --fastskip     # 跳过所有非关键检查，最快交付
/夯 登录功能       # 正常流程，按规则触发检查
/夯 --thorough     # 全量检查，适合高敏模块
```

#### 铁律 2：每个环节计时，超时自动放行

对于 Plan 输出、review 等环节设定硬超时：

- Plan 生成: 最长 15s（包括写入 plan.md）
- Code Review: 最长 30s（超过则输出当前已发现的 issues 就过，不卡住）
- Test Generation: 最长 20s（超过则先提交代码，测试异步补充）

#### 铁律 3：增量优先，全量兜底

- Review 只 diff 变更部分，不重新 review 整个代码库
- 测试只跑受影响模块，不跑全量（全量回归在 Stage 4）
- Spec 更新只 diff 变更字段，不重写整个 Spec

#### 铁律 4：Measured，Not Guessed

在 settings.json 中埋点记录每个环节耗时：

```json
{
  "telemetry": {
    "enabled": true,
    "metrics": {
      "plan_generation_ms": 1234,
      "review_duration_ms": 5678,
      "review_rounds": 1,
      "test_generation_ms": 2345
    }
  }
}
```

每月审视指标：**平均交付时间是否因优化而缩短？如果否，回退优化。**

### 6. 优先级 Roadmap

| 优先级 | 优化项 | 理由 | 预估工作量 |
|--------|--------|------|-----------|
| **P0** | Spec 增强 — 加入验收条件 + 边界情况字段 | 最高 ROI：减少方向性返工，降低 Agent 瞎猜概率。改动最小（模板 + 示例），收益立竿见影 | 1-2 天（模板修改 + kf-spec 适配） |
| **P0** | Review 条件触发重审（P0/P1 缺陷 → 自动重审） | 一键改动：在 kf-code-review-graph 的输出判断逻辑加个 if 分支。不需要重审的 case 零成本 | 0.5-1 天（review 输出分类 + 条件触发） |
| **P0** | 查阅式 Plan 输出（非阻塞） | 替代阻塞式审批，保留人类掌控感但去掉了等待瓶颈。改动小但影响面广 | 1 天（kf-spec + `/夯` 适配 plan.md 输出） |
| **P1** | 单元测试随代码生成 | 长期收益高，但改动涉及编码阶段的全流程。需要注意不拖慢代码生成速度 | 2-3 天（编码 pipeline 的 test 钩子） |
| **P1** | 关键路径集成测试前移 | 对核心模块有明显价值，但需要定义"关键路径"的判定规则 | 2-3 天（规则定义 + pipeline 适配） |
| **P1** | 效率监控埋点 | 没有数据就没有优化决策。但埋点系统需要从 0 搭，且不能影响主流程 | 2-3 天（中间件 + 存储 + 报告） |
| **P2** | 多轮 review 全量支持（非条件触发） | 收益有限，成本高。只在安全敏感场景有需求，大部分场景不需要 | 3-5 天（循环机制 + 最大轮次控制 + 成本监控） |
| **P2** | 全量测试前移 | 高风险低收益，建议观察 ListenHub 的长期实践效果再决定是否跟 | 5-10 天（测试框架适配 + pipeline 改造 + 稳定性治理） |

#### 执行顺序

```
Phase 1（快赢）:
  P0: Spec 增强 + 条件 review + 查阅式 Plan
  → 收益：减少方向性返工，消除审批等待

Phase 2（夯实）:
  P1: 单元测试伴随 + 关键路径测试 + 效率监控
  → 收益：提升代码质量基数，建立度量体系

Phase 3（观望）:
  P2: 多轮 review + 全量测试前移
  → 先等 ListenHub 跑出数据，看是否值得跟进
```

### 风险提示

1. **过度工程的风险**：Spec 增强最容易滑向过度工程。验收条件 + 边界情况是甜区，再加字段就是在增加 Spec 写作成本而不是编码成本。**守住的底线：Spec 不能比代码长。**

2. **审批惰性**：如果用了"查阅式 Plan"，人类可能从不查阅，导致 Agent 跑偏到无法挽回的程度才被发现。需要配合"关键里程碑提示"机制——在 Plan 中的重要节点（如数据库变更、API 设计）主动推送通知，而不是等人类自己来看。

3. **测试维护成本被低估**：单元测试前移是好的，但要警惕"测试代码量超过业务代码量"的反常。Agent 生成测试是好事，但需要有机制控制测试规模（每个函数最多 3 个测试用例，边界+正常+异常，多了就是过度测试）。

4. **效率数据被误读**：埋点数据容易引起"唯速度论"——谁快谁好。但速度和质量是 trade-off。监控的目的是发现问题，不是 KPI 考核。如果团队因为埋点数据压力而跳过 review，就是好心办坏事。

5. **ListenHub 的适用性边界**：ListenHub ColaOS 的工作流是 UP 主单人 + 20 个仓库的场景，而我们的是多技能团队协作。直接对标 Plan 审批 / 多轮 review 可能导致水土不服。**借鉴理念，不照搬流程。**

**一句话总结蓝队立场：先干了再说，偏了再调，别在开工前把时间花在"讨论怎么干活"上。**
