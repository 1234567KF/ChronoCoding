**分析**：两者对 `ls` 都有极好压缩效果。lean-ctx 略优（84% vs 76.5%），叠加后达到 93.5%。RTK 保留了文件大小信息（`_index.json 5.7K`），lean-ctx 纯目录名更精简。

## 五、关键发现

### 5.1 核心结论

1. **lean-ctx 综合节省远高于 RTK**：平均 72% vs 25.7%，差距主要来自 git log 和大文件场景
2. **RTK 对 ls 类操作高效**：76.5% 节省率，接近 lean-ctx 的 84%。适合频繁 ls 的开发者
3. **大文件场景 lean-ctx 占绝对优势**：MCP 的引用模式返回引用而非内容，从根本上消除大文件 token 消耗
4. **叠加效果在 ls 场景显著**：C4-S1 达 93.5%（叠加增益 17.5 ppts），其他场景叠加增益有限
5. **RTK 对结构化输出（git log oneline、文件内容）压缩力不足**：多个场景 0% 压缩

### 5.2 叠加策略建议

| 场景 | 推荐配置 | 理由 |
|------|---------|------|
| 目录浏览/文件操作 | RTK + lean-ctx | 叠加 93.5% 高收益 |
| git 操作 | 仅 lean-ctx | RTK 无贡献，避免冗余 hook 开销 |
| find/grep 搜索 | 仅 lean-ctx | RTK 增益 < 6ppts |
| 大文件读取 | 仅 lean-ctx | RTK 完全无贡献 |
| 构建/测试输出 | RTK + lean-ctx | RTK 对 test/build 有专用压缩 |

## 七、建议

1. **优先启用 lean-ctx**：单工具即达 72% 平均节省，覆盖全场景
2. **RTK 作为 ls/test/build 场景补充**：在 lean-ctx 基础上叠加 ls 场景再省 17.5 ppts
3. **无需追求全叠加**：S2/S3/S4 场景叠加增益 < 6ppts，额外 hook 开销不值
4. **lean-ctx 的 ctx_read 大文件模式是关键武器**：map/reference 模式根本上消除大文件 token
