### 核心策略

1. **全局 System Prompt 统一前缀** — 各技能共享同一 system prompt 前缀段落
2. **长上下文预热** — 先发一次带长文档的请求让缓存落盘，再发真正问题
3. **缓存命中率追踪** — `usage.prompt_cache_hit_tokens` 纳入 token-tracker 可视化

## 四、三角色架构

```
┌─────────────────────────────────────────────┐
│                 Team Lead                    │
│          （任务理解 + 拆解 + 调度）           │
├──────────┬──────────────────┬────────────────┤
│ 节流者    │    协作者        │   监测者        │
│ (Saver)  │  (Collaborator)  │  (Monitor)     │
│          │                  │                │
│ • 缓存    │ • 多 Agent 编排  │ • Token 追踪   │
│ • 路由    │ • 角色分配       │ • 成本审计     │
│ • 成本    │ • 并行调度       │ • 性能监控     │
│ • 压缩    │ • 结果融合       │ • 报警决策     │
└──────────┴──────────────────┴────────────────┘
```

### 节流者（Saver）
- KV Cache 公共前缀预热
- 自动检测缓存命中率，低于阈值时调整 system prompt
- 模型智能路由强化（强制不用 pro 的场景）
- Context 压缩（lean-ctx 集成）
- 成本预算控制

### 协作者（Collaborator）
- 对标 Qoder Experts 的 Team Lead 角色
- 动态征召专家（按任务类型拉起不同组合）
- 用户随时介入修正方向
- 结果整合 + 冲突消解

### 监测者（Monitor）
- 实时任务面板
- 缓存命中率仪表盘
- 各 Agent 成本实时统计
- 自动标记异常模式

## 五、交互方式升级

当前：用户说一句话 → 系统开始执行 → 用户只能等待结果。

目标：
```
用户 → Team Lead 即时响应（理解+拆解+计划）
     → 用户确认/调整
         → 并行征召专家执行
             → 用户可随时介入修正
                 → 最终交付
```
