## 二、问题1：监控体系能否支持可重复稳定测试？

### 当前skill-monitor v3.0能力

| 能力 | 状态 | 问题 |
|------|------|------|
| PreToolUse hook捕获技能调用 | ✅ | 仅主进程生效，子代理不继承 |
| Token追踪（input/output/cache） | ✅ | 数据来自Claude API响应 |
| 缓存命中/未命中 | ✅ | v3新增 |
| 成本估算 | ✅ | 基于定价表 |

### 核心问题：hook只在主进程生效

三队/夯实测发现：**子代理不继承settings.json的hooks**，skill-monitor无法被动追踪子代理行为。

### 可重复测试方案

**方案A：标准化测试脚本（推荐）**

设计一套固定的测试用例，每次运行相同输入，对比输出：

| 测试项 | 测试脚本 | 重复性 |
|--------|---------|--------|
| lean-ctx压缩率 | lean-ctx -c "git log --oneline -20" | ✅ 稳定（同一repo结果一致） |
| model-router切换 | 给出"计划+执行"两阶段任务 | ✅ 可设计触发条件 |
| .claudeignore效果 | 对比有/无ignore的context大小 | ✅ 稳定 |
| /compact效果 | 固定对话长度后执行/compact | ⚠️ 取决于对话内容 |
| Caveman输出压缩 | 安装后对比同任务输出token | ✅ 稳定 |

**方案B：主动式监控（补充）**

在spawn子代理前后主动调用skill-monitor.log：

```
主进程: skill-monitor.log spawn_start team=red
  → spawn子代理
  → 子代理完成
主进程: skill-monitor.log spawn_end team=red output=14KB duration=124s
```

这样即使子代理内部没有hook，主进程也能记录完整的调度+产出数据。

**结论**：可以做到可重复稳定测试，但需要：
1. 写一套标准化测试脚本（而非依赖hook被动捕获）
2. 主进程主动记录spawn前后的关键指标
3. 每次测试用相同输入+相同环境，对比delta
