## 用例1：CLI 输出压缩（lean-ctx）

### 测试设计

| 项目 | 说明 |
|------|------|
| 测试目标 | 验证 `git status`、`npm test` 等命令的输出压缩效果 |
| 测试方法 | 对比原始输出 char 数 vs lean-ctx 压缩后 char 数 |
| 数据来源 | `03-搜索结果/lean-ctx评测数据.md` 中的实测数据 |

### 实测数据（来自已有评测）

| 命令 | 原始输出 | lean-ctx 输出 | 压缩率 | 来源 |
|------|---------|--------------|-------|------|
| `vitest run` | 102,199 chars | 377 chars | **-99.6%** | Terminal-Bench 实测 |
| `npm test` | 25,000 chars | 2,500 chars | **-90%** | 社区反馈 |
| `cargo test` | 25,000 chars | 2,500 chars | **-90%** | 社区反馈 |
| `pytest` | 20,000 chars | 5,000 chars | **-75%** | 社区反馈 |
| `git status` | 7,500 chars | 1,500 chars | **-80%** | 社区反馈 |
| `ls / tree` | 2,000 chars | 400 chars | **-80%** | 社区反馈 |

### 分析

**高压缩率命令类型**：
- 测试输出（vitest/npm test/cargo test）→ 极高压缩（90%+），因为测试框架输出冗长
- Git 操作（status/diff/log）→ 高压缩（75-80%），因为输出结构化
- 文件列表（ls/tree）→ 高压缩（80%），因为输出重复

**低压缩率命令类型**：
- 编译输出（gcc/clang）→ 低压缩，因为错误信息需要完整
- 动态内容（curl/wget）→ 中压缩，因为内容随机

### Token 节省估算

假设每次开发 session 执行 50 次 CLI 命令，平均每次命令输出 10,000 chars：

```
原始 token 消耗 = 50 × 10,000 chars = 500,000 chars ≈ 500K tokens
lean-ctx 压缩后 = 50 × 1,000 chars = 50,000 chars ≈ 50K tokens
节省率 = 90%
```

## 用例2：模型路由（kf-model-router）

### 测试设计

| 项目 | 说明 |
|------|------|
| 测试目标 | 验证 Pro 计划 + Flash 执行的质量等价性和 token 节省 |
| 测试方法 | 同一任务对比 Pro-only vs Pro+Flash 的输出质量和 token |
| 当前状态 | kf-model-router 在本项目已配置 |

### 估算数据（基于定价）

| 模型 | 输入价格 | 输出价格 | 典型任务消耗 |
|------|---------|---------|-------------|
| Pro (sonnet-4) | $3/MTok | $15/MTok | 计划阶段：5K in + 2K out |
| Flash (sonnet-4 mini) | $0.15/MTok | $0.6/MTok | 执行阶段：20K in + 8K out |
| **节省率** | **95%** | **96%** | 综合约 95% |

### 质量验证原则

> ⚠️ 节省 token 的前提是输出质量不能显著下降

质量验证方法：
1. 简单任务（搜索、格式化）：Flash 直接执行，质量应与 Pro 相近
2. 复杂任务（架构设计、评审）：Pro 分析 → Flash 执行，定期抽检质量
3. 关键任务（安全审计、PR评审）：保持 Pro 执行，避免质量风险

|------|------|
| 测试目标 | 验证 <3 文件跳过 spawn 的准确性和回调替代轮询的节省 |
| 测试方法 | 统计多 Agent 任务中的 spawn 次数和轮询开销 |
| 当前状态 | ccp-smart-dispatch.cjs 已配置 |

### 估算节省

| 场景 | 无 CCP | 有 CCP | 节省 |
|------|--------|--------|------|
| 3 文件任务（应跳过 spawn） | 15K token | 0（跳过） | **~15K token** |
| 10 文件任务（应 spawn） | 15K token | 15K token + 回调注入 | 轮询节省 80-97% |
| 100 次轮询检查 | ~100 × 500 = 50K token | 回调 0 + 1 次确认 | **~97%** |

### 判断逻辑（来自 ccp-smart-dispatch.cjs）

```javascript
// 跳过 spawn 的条件（<3 文件）
const shouldSkipSpawn = fileCount < 3 && !hasComplexDependencies;

// 回调替代轮询
const useCallback = agentSupportsCallback && estimatedPollCost > callbackOverhead;
```
