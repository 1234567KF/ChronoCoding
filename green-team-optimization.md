## 绿队方案 — 准与测的准

### 总体立场
质量不是靠流程堆出来的，是靠数据测出来的。每个环节必须有可量化的准入/准出标准，每道门必须知道什么时候开、什么时候关、开了之后效果如何。不信任一次输出，但也不盲目叠流程——用数据决定流程深度。

---

### 1. Plan 审批关卡

#### 现状诊断
当前 `/夯` 和 `kf-spec` 无显式审批，Agent 拆解任务后直接执行。风险在于：Agent 的方向偏差要到 Stage 3/4 才能被发现，此时返工成本已很高。

#### 数据驱动的条件审批模型

**不是所有任务都需要人类审批。关键问题是：什么情况下人类的 30 秒审批能节省 Agent 的 30 分钟返工？**

##### 需要审批的任务类型（基于数据特征）

| 风险等级 | 判定条件 | 示例 | 审批方式 | 预计节省返工 |
|----------|----------|------|----------|-------------|
| **高风险 (R3)** | 跨 >3 模块 / 涉及安全/支付/数据隐私 / 新建 >10 个文件 | 用户认证重构、支付集成 | 人类必须审批 Plan | 预估可减少 60-80% 的重大方向性返工 |
| **中风险 (R2)** | 跨 2-3 模块 / 新增 5-10 个文件 / 修改已有 API 契约 | 新增业务列表页、修改数据模型 | 条件审批：自动标注高风险点，人类可跳过 | 预估减少 30-50% 的中等返工 |
| **低风险 (R1)** | 单模块内 / 修改 <5 个文件 / 纯新增无修改 / bug fix 有复现步骤 | 简单 CRUD、样式调整、文档更新 | 自动通过，记入审计日志 | 审批成本接近 0 |

##### 风险评分公式

```
RiskScore = w1 × module_cross_count + w2 × new_file_count + w3 × modified_file_count
          + w4 × security_sensitive_flag + w5 × api_contract_changed_flag
          + w6 × test_coverage_before_flag

Security-sensitive file 列表: 包含 auth/payment/password/token/key/secret 等关键词的文件
Api contract changed: 修改了 routes/controllers/interfaces 定义
```

建议权重（可从历史数据校准）：
- w1=2.0, w2=0.3, w3=0.5, w4=5.0, w5=3.0, w6=-2.0（有测试覆盖降低风险分）

##### 审批有效性的度量

审批本身也需要"测的准"。引入以下指标：

| 指标 | 公式 | 目标 | 说明 |
|------|------|------|------|
| **审批通过率** | 通过的 Plan / 总 Plan | — | 基线指标，极端值告警 |
| **审批有效拦截率** | 审批发现方向问题数 / 审批总次数 | >30% | 如果拦截率 <10%，说明审批太松或不需要走审批 |
| **审批误拦率** | 审批拒绝但实际正确执行且无缺陷的 Plan / 审批拒绝总数 | <10% | 如果误拦率 >20%，说明审批标准有问题 |
| **审批成本** | 人类审批耗时 + 等待时间 / Plan | <2 分钟 | 单个 Plan 审批超过 5 分钟说明流程太重 |

##### 具体改造建议

在 `kf-multi-team-compete` 的 Pre-Stage 和 `kf-spec` 的 Plan 生成之间插入条件审批关卡：

```
用户输入
  ↓
kf-alignment (需求对齐)
  ↓
kf-spec (生成 Spec)
  ↓
Agent 自动拆解 Plan + 自动标注风险等级
  ↓
┌─ 风险评分 <= 阈值 (R1) → 自动通过，记入日志 ──────────────┐
│                                                           ↓
├─ 风险评分 中 (R2) → 输出 Plan + 风险标注 → 用户 30 秒确认 → 执行
│                                                           ↓
└─ 风险评分 高 (R3) → 输出 Plan + 风险报告 → 必须人类审批 ──→ 执行
                                                           ↓
                                                    开始编码
```

审批输出的 Plan 文件应包含：
- 任务拆解列表（每个任务标注文件范围和预估改动量）
- 风险评分明细（哪些维度触发了高分）
- 变更影响域（受影响模块、API、数据模型）
- 建议的测试覆盖类型（单元/集成/E2E）

### 2. Review-Fix 反馈环

#### 现状诊断
`kf-code-review-graph` 当前是一次性调用，输出 review 报告后流程继续。但 ListenHub 的经验表明：**AI 的第一次输出不可信，需通过 review-fix 循环逼近质量标准**。

#### 条件触发的多轮 Review 模型

**不是所有 review 问题都需要走完整循环。关键问题是：什么指标触发下一轮？**

##### 触发条件矩阵

| 条件 | 数据指标 | 阈值 | 动作 |
|------|----------|------|------|
| **P0 问题存在** | 安全漏洞/数据损坏/逻辑错误 | 数量 > 0 | **强制第二轮 review**，修复后必须重审 |
| **P1 问题密度** | P1 issues / KLOC | > 3 个/KLOC | **触发第二轮 review** |
| **整体质量评分** | 综合评分（0-10） | < 6.0 | **触发第二轮 review** |
| **修复回溯率** | 上一轮的 P0/P1 在本轮重新出现 | > 20% | **触发架构级 review**（非代码级） |
| **通过条件** | 综合评分 + 无 P0 + P1 低密度 | >= 7.0 且 P0=0 且 P1 < 2/KLOC | **一次通过，无需循环** |

##### Review 问题分级

| 等级 | 定义 | 第二轮要求 | 修复时限 |
|------|------|-----------|---------|
| **P0 阻断性** | 安全漏洞、数据损坏、核心逻辑错误 | 强制，修复后必须由重审通过 | 立即 |
| **P1 严重** | 功能偏离、性能风险、边界未处理 | 密度 > 阈值则重审 | 本轮 |
| **P2 一般** | 代码风格、可读性、未使用 import | 不触发重审，记入代码债 | 可延后 |
| **P3 建议** | 优化建议、备选方案 | 不触发重审 | 可延后 |

##### 反馈环的执行流程

```
Round 1: kf-code-review-graph 全量 review
  → 输出问题列表 + 质量评分
  → 计算问题密度（issues/KLOC）
  ↓
┌─ 满足通过条件？(评分 >= 7.0, P0=0, P1 密度 < 2/KLOC)
│  是 → 结束，输出 review 报告（含待处理的 P2/P3 代码债）
│  否 → 进入修复阶段
│       ↓
│      Agent 自动修复 P0/P1 问题
│       ↓
│      Round N: 增量 review（只审修复的部分 + 回归检查已通过部分）
│       ↓
│      ┌─ 新引入 P0？→ 继续循环（最多 3 轮）
│      │  新引入 P1 密度 > 阈值？→ 继续循环
│      │  达到最大轮次 3 轮？→ 升级为人类介入
│      └─ 通过？→ 结束
```

##### 数据指标监控

| 指标 | 公式 | 用途 |
|------|------|------|
| **首次通过率** | Round 1 通过的 review / 总 review | 衡量 Agent 代码质量基线 |
| **平均轮次** | 总 review 轮次 / 总 review | 衡量修复效率 |
| **问题衰减率** | (Round N P0+P1) / (Round 1 P0+P1) | 衡量反馈环收敛速度 |
| **P0 逃逸率** | 上线后发现的 P0 / 上线前发现的 P0 | 衡量 review 覆盖率 |
| **代码债累积率** | P2+P3 问题 / KLOC / 时间 | 衡量技术债务增长速度 |

### 3. Spec 结构化增强

#### 现状诊断
当前 `kf-spec` 生成的 Spec 偏概要，Agent 执行时有大量"临场发挥"空间。根据 ListenHub 的经验，Spec 应该是 **AI 可执行的结构化蓝图**，而非人类阅读的散文式文档。

#### 增强的 Spec 模板

```markdown
# [功能名称] - Spec

## 1. 概述
{一句话描述功能定位}

## 2. 功能列表
### F1: [功能名]
- **描述**: {具体描述}
- **优先级**: P0/P1/P2
- **预估改动量**: {文件数/行数}

## 3. 验收条件 (Acceptance Criteria)
### AC1: [验收条件名称]
- **条件**: {具体的行为描述}
- **验证方式**: {单元测试 / 集成测试 / E2E / 人工验收}
- **Pass 条件**: {明确的通过标准}
- **Fail 条件**: {明确的失败标准}
- **示例**:
  - Given {前置条件}
  - When {操作}
  - Then {期望结果}

### AC2: ...

## 4. 边界情况 (Edge Cases)
| # | 场景 | 预期行为 | 测试类型 |
|---|------|---------|---------|
| E1 | 输入为空 | {行为} | 单元测试 |
| E2 | 并发请求 | {行为} | 集成测试 |
| E3 | 网络超时 | {行为} | 集成测试 |
| E4 | 权限不足 | {行为} | E2E 测试 |

## 5. 测试策略 (Test Strategy)
| 测试层 | 覆盖目标 | 覆盖率要求 | 执行时机 |
|--------|---------|-----------|---------|
| 单元测试 | 核心逻辑、边界条件 | >85% line coverage | 每次提交 |
| 集成测试 | API 契约、数据流 | >70% API endpoints | 模块完成 |
| E2E 测试 | 关键用户路径 | 所有 P0 AC | Stage 3 |
| 安全测试 | 认证/授权/注入 | 所有安全敏感文件 | Stage 2 |

## 6. 依赖关系 (Dependencies)
| 依赖 | 类型 | 影响 |
|------|------|------|
| 前置功能 | 内部模块依赖 | {功能名}完成前无法开始 |
| 外部服务 | API/第三方 | {服务名}不可用时的降级策略 |
| 数据依赖 | 数据模型 | 需要先迁移 {表名} |

## 7. 数据模型变更
{只在此功能需要修改数据模型时填写}
- 新增字段: {字段名}: {类型} — {说明}
- 修改字段: {字段名}: {旧类型→新类型} — {迁移说明}
- 删除字段: {字段名} — {数据清理说明}
```

#### 结构化字段的验证规则

每个结构化字段可以**自动验证完整性**，独立于具体功能内容：

```javascript
// Spec 完整性检查规则示例
const specValidationRules = {
  acceptanceCriteria: {
    required: true,
    minCount: 1,  // 每个功能至少 1 个 AC
    each: {
      passCondition: { required: true, type: 'binary_statement' },
      failCondition: { required: false, type: 'binary_statement' },
      verificationMethod: { required: true, enum: ['unit', 'integration', 'e2e', 'manual'] }
    }
  },
  edgeCases: {
    required: false,
    recommended: 'number_of_ACs >= 2',  // 如果 AC >= 2，建议至少 1 个边界情况
  },
  testStrategy: {
    required: true,
    coverageTargets: {
      unit: { min: 70 },  // 单元测试最低覆盖率
      integration: { min: 50 },
      e2e: { min: 30 }
    }
  },
  dependencies: {
    required: function(spec) { return spec.moduleCrossCount > 1; }
    // 跨模块时必须声明依赖
  }
}
```

#### Spec 质量评分

```markdown
## Spec 质量评分卡（自动生成）

| 维度 | 权重 | 评分标准 | 得分 |
|------|------|---------|------|
| AC 完整性 | 30% | AC 覆盖所有功能，每个 AC 有 pass/fail | /100 |
| AC 可验证性 | 25% | AC 的 pass/fail 是二值可测的 | /100 |
| 边界覆盖 | 20% | 边界情况数量 >= AC 数量的 50% | /100 |
| 测试策略 | 15% | 明确每层覆盖目标和工具 | /100 |
| 依赖完整性 | 10% | 依赖关系完整标注 | /100 |

**综合评分**: {score}/100
**是否可执行**: {是/否（评分 >= 70 可执行）}
```

当 Spec 评分 < 70 时，**禁止进入编码阶段**，Agent 必须补充完整后再提审。

### 4. 测试前移

#### 现状诊断
`kf-browser-ops` 在 Stage 3 才介入，这意味着代码编写和集成测试之间有一个空窗期。绿队主张**测试不是在某个阶段做的一件事，而是贯穿始终的保障机制**。

#### 分层测试矩阵

| 测试层 | 工具/技能 | 覆盖目标 | 覆盖率要求 | 执行时机 | 触发条件 |
|--------|----------|---------|-----------|---------|---------|
| **L0: 静态分析** | kf-code-review-graph + ESLint/TypeScript | 语法、类型、规范 | 100% 代码 | Pre-commit | 每次文件变更 |
| **L1: 单元测试** | vitest/jest (由 kf-spec 生成骨架) | 核心函数、工具类、数据模型 | >85% line coverage | Stage 1 (编码) | 每完成一个功能模块 |
| **L2: API 集成测试** | supertest / k6 | API 端点、数据流、错误处理 | >70% endpoints | Stage 2 (集成) | 后端路由定义完成 |
| **L3: UI 组件测试** | Vitest + Happy DOM / Testing Library | 组件渲染、交互、状态 | >60% components | Stage 2 (集成) | UI 原型生成后 |
| **L4: E2E 测试** | kf-browser-ops (Playwright) | 关键用户路径 | All P0 AC | Stage 3 | 功能开发完成 |
| **L5: 回归测试** | 全量 L1-L4 测试套件 | 防止回归 | 100% 已通过用例 | Pre-merge / Stage 4 | Merge 前必须全部通过 |
| **L6: 安全测试** | kf-code-review-graph 安全规则 + 可扩展 | 认证/授权/注入/XSS | 安全敏感文件 100% | Stage 2 + Stage 4 | 检测到安全敏感文件变更 |

#### 分层准入/准出标准

```
Stage 1 (编码):
  准入: Spec 评分 >= 70
  过程: 每完成一个功能 → L0 + L1 通过
  准出: L0/L1 通过率 100%, line coverage > 85%

Stage 2 (集成):
  准入: Stage 1 准出
  过程: 每完成一个模块 → L2 + L3 通过
  准出: L2 endpoints > 70%, L3 components > 60%

Stage 3 (E2E):
  准入: Stage 2 准出
  过程: L4 全量执行
  准出: P0 AC 全部通过 (E2E)

Stage 4 (回归 + 审查):
  准入: Stage 3 准出
  过程: L5 + L6 全量
  准出: 所有测试通过 + Review 评分 >= 7.0
```

#### 测试数据驱动优化

```javascript
// 测试有效性度量
const testMetrics = {
  // 测试覆盖质量（非数量）
  mutationScore: 0,  // 变异测试分数，目标 > 70%
  defectDetectionRate: 0,  // 缺陷检测率，目标 > 90%
  
  // 测试效率
  avgSpeedPerTest: 0,  // 毫秒/测试
  totalRunTime: 0,     // 总运行时间
  
  // 回归测试健康度
  regressionPassRate: 0,       // 回归通过率
  flakyTestRate: 0,            // 不稳定测试比例，目标 < 5%
  testMaintenanceCost: 0,      // 每次迭代需要修改的测试比例
}
```

### 5. 质量保障体系

#### 核心理念：数据驱动，非流程堆砌

多一个审批环节不等于质量提升。质量保障体系的本质是：**在每个决策点提供足够的数据，让决策者（人或 AI）做出有依据的判断，并闭环验证决策效果。**

#### 质量信号体系

```
                      ┌──────────────────────┐
                      │    质量信号控制台      │
                      │  (数据仪表盘，非流程)  │
                      └──────────┬───────────┘
                                 │
          ┌──────────────────────┼──────────────────────┐
          │                      │                      │
   ┌──────▼──────┐      ┌──────▼──────┐      ┌──────▼──────┐
   │  Spec 质量    │      │  代码质量     │      │  测试质量     │
   │  - AC 完整性  │      │  - Review评分  │      │  - 覆盖率     │
   │  - 边界覆盖   │      │  - 问题密度   │      │  - 缺陷检测率  │
   │  - 可验证性   │      │  - 修复率     │      │  - 回归稳定度  │
   │  - 依赖明确   │      │  - 代码债累积  │      │  - 执行速度    │
   └──────┬──────┘      └──────┬──────┘      └──────┬──────┘
          │                      │                      │
          └──────────────────────┼──────────────────────┘
                                 │
                      ┌──────────▼───────────┐
                      │   质量门控决策引擎      │
                      │                      │
                      │  R1: 自动通过          │
                      │  R2: 条件通知          │
                      │  R3: 阻断 + 人类介入   │
                      └──────────────────────┘
```

#### 质量门控阈值（初始建议，需通过数据校准）

| 质量维度 | 绿灯 (自动通过) | 黄灯 (通知) | 红灯 (阻断) |
|----------|---------------|------------|------------|
| Spec 评分 | >= 80 | 60-79 | < 60 |
| Review 评分 | >= 7.5 | 6.0-7.4 | < 6.0 |
| 单元测试覆盖率 | >= 85% | 70-84% | < 70% |
| E2E 通过率 | 100% | 95-99% | < 95% |
| 回归测试通过率 | 100% | 98-99% | < 98% |
| 问题密度 (P0+P1) | 0 / KLOC | < 2 / KLOC | >= 2 / KLOC |
| 不稳定测试率 | < 3% | 3-10% | > 10% |

#### 质量闭环

```
质量门控阻断
  ↓
自动诊断: 什么维度不达标？趋势是变好还是变差？
  ↓
根因分析: 是 Spec 问题 / 代码问题 / 测试问题？
  ↓
定向修复: 根据根因选择修复策略
  ↓
重新验证: 通过后才放行
  ↓
数据回写: 更新质量基线，调整阈值
```

#### 关键区别：数据驱动 vs 流程驱动

| | 纯流程驱动 | 数据驱动（绿队主张） |
|--|-----------|-------------------|
| 规划审批 | 所有任务都要审批 | 只有风险评分 > 阈值的需要审批 |
| Code Review | 每次必审固定轮次 | 根据质量评分和问题密度条件触发 |
| 测试要求 | "要有测试" | 每层有具体覆盖率目标和有效性度量 |
| 质量门控 | 固定步骤列表 | 动态阈值 + 趋势分析 |
| 改进方式 | 加更多流程环节 | 分析数据，优化阈值和触发条件 |
| 人力成本 | 线性增长 | 亚线性（数据越准，自动化越高） |

### 6. 优先级 Roadmap

| 优先级 | 优化项 | 理由 | 预估工作量 |
|--------|--------|------|-----------|
| **P0** | **Plan 条件审批关卡** | 当前最显著的缺失——方向偏差的返工成本远高于编码成本。引入风险评分模型可拦截 60-80% 的重大方向性返工，且 R1 任务自动放行不增加等待。 | 3-5 天（风险评分模型 + 审批交互 + Agent 流程改造） |
| **P0** | **Review-Fix 反馈环** | Review 不走闭环等于没审。P0 问题强制重审是质量底线。引入后可预期首次通过率从当前估算的 ~40% 提升至 >70%。 | 3-5 天（循环控制 + 评分模型 + 修复 Agent） |
| **P1** | **Spec 结构化增强** | Spec 是质量的第一道闸门，当前太概要导致大量执行偏差。增强后 Spec 质量评分 < 70 禁止编码。不是所有功能都需要完整模板——P0/P1 功能必须完整，P2 可以简化。 | 4-6 天（模板设计 + Spec 校验器 + 现有 kf-spec 改造） |
| **P1** | **L1 单元测试前移** | 单元测试在 Stage 1 编码阶段并行生成和执行，是最低成本最高回报的测试层。优先于集成和 E2E。 | 2-4 天（集成 vitest + 模板 + 与 kf-spec AC 关联） |
| **P2** | **L2-L3 集成测试 + 组件测试** | 在 Stage 2 集成阶段自动化执行，需要 API 路由定义和组件结构就绪。 | 4-6 天（API 测试框架 + 组件测试框架） |
| **P2** | **质量信号控制台** | 可视化仪表盘，显示 Spec/代码/测试三个维度的实时数据。初期可简化为每次迭代输出的质量报告 markdown。 | 3-5 天（数据采集 + 指标计算 + 报告生成） |
| **P2** | **阈值自动校准** | 基于历史数据自动调整质量门控阈值，避免人工设定不合理的硬编码值。 | 4-6 天（历史数据收集 + 统计分析 + 动态阈值模型） |
| **P2** | **L4-L6 测试全量自动化** | E2E + 回归 + 安全测试全量自动化执行，作为 Stage 4 的准出门控。 | 5-8 天（Playwright 集成 + 回归套件 + 安全规则库） |

### 风险提示

1. **过度审查化风险**：条件审批如果阈值设得太高（太多 R3），会让审批成为瓶颈。必须通过第一阶段的审批数据校准阈值，核心指标是"审批有效拦截率 > 30%"和"审批误拦率 < 10%"。如果拦截率太低，说明阈值太高，应降低风险等级。

2. **Review 循环发散风险**：Review-Fix 循环可能不收敛（Agent 修复引入新问题，新问题触发新一轮）。必须设硬上限：最大 3 轮，超限则升级为人类介入。同时关注"问题衰减率"指标——如果 Round 2 的 P0+P1 数量不低于 Round 1 的 50%，说明修复策略有系统性问题。

3. **测试维护成本失控风险**：前移测试可能产生大量"脆性测试"（brittle tests）——频繁因实现细节变更而失败。必须监控"不稳定测试率"和"测试维护成本"两个指标。如果 flaky test rate > 10%，需要暂停测试前移，先做测试基础设施加固。

4. **Spec 过度模板化风险**：结构化模板如果太死板，会让 Agent 花更多时间填模板而非理解需求。建议：P0/P1 功能用完整模板，P2 功能用简化版（只需 AC + 测试策略）。模板本身也要通过使用数据持续优化——删除那些频繁留空或无用的字段。

5. **数据滞后风险**：质量门控依赖的数据如果采集延迟（如覆盖率报告在测试执行后 10 分钟才生成），会导致门控判断滞后。必须确保关键指标（review 评分、测试通过率）是同步可得的，非关键指标可以异步。

6. **Human-in-the-loop 疲劳风险**：如果每个 Plan 都要人类审批，人类会快速疲劳并"习惯性通过"，使审批形同虚设。必须用条件审批（只审高风险的）来保护人类的注意力资源——**人类只在数据说"这里有风险"时才介入**，而不是在每个环节都介入。

7. **绿队的自我约束**：绿队主张"数据驱动"，但数据本身也可能误导。初期阈值缺乏历史数据支撑，应采用"渐进式"策略：先设较宽松的阈值（避免过度阻断），积累 2-4 周数据后再做精确校准。
