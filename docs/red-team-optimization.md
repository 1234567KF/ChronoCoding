## 红队方案 — 稳字当头

### 总体立场

ListenHub ColaOS 验证了一个核心洞见：**AI 执行的质量上限 = 人类把关的颗粒度**。Spec/Plan 人类确认 + 多轮 Review 闭环 + 测试全贯穿，这三件事共同构成了"稳"的落地支柱。我们的 kf- 体系在"快"和"省"上已有优势，但"稳"的工程化手段仍需补齐——以下方案的目标是：**引入可控的摩擦力，消除失控的返工成本**。

---

### 1. Plan 审批关卡

**立场：所有任务必须经过人类 Plan 审批。**

**为什么 Plan 审批是必要的**

- **消除方向性浪费**：Agent 拆解 Plan 时可能对优先级排序错误、粒度不适当、遗漏关键依赖。未经审批就执行，发现方向错误时所有计算都已沉没。Plan 审批的成本是分钟级，错误执行的成本是小时级 + token 浪费。
- **防止 scope creep**：Agent 倾向于过度拆解或遗漏非功能性需求。人类审批可以校准 scope。
- **建立信任基线**：当前用户对自动执行的 Plan 没有 visibility，出问题后信任修复成本很高。

**交互设计**

```
kf-spec → Plan 生成 → [PLAN_APPROVAL 关卡] → 三队并发执行
                            │
                    ┌───────┴────────┐
                    │  展示 Plan       │
                    │  a) approve     │
                    │  b) edit        │
                    │  c) reject      │
                    └────────────────┘
```

- plan.md 固化输出到 `.claude/plans/current-plan.md`，用户可直接编辑
- 三键操作：approve / edit / reject，避免复杂表单
- 用户可在 `/夯` 命令中加 `--auto` 跳过审批（适合紧急修复）

**效率优化**：
- 智能默认批准：Plan 改动量 < 3 任务且与历史匹配 → 自动批准
- 并行展示：Plan 生成的同时预热 Agent 环境
- 分阶段审批：大任务只审批第一阶段，后续滚动审批
- 学习模式：反复出现的同类任务自动预批准

### 2. Review-Fix 反馈环

**立场：必须改为 review→fix→re-review 循环，上限 3 轮。**

**为什么一次 Review 不够**

- 第一轮 Review：捕获约 60-70% 的问题
- 第二轮 Review：捕获约 20-25%（修复引入的新问题、深层设计问题）
- 第三轮 Review：捕获约 5-10%（边界情况、一致性细节）

单轮 Review 的盲区：Agent 修复问题本身可能引入新问题，不经过 re-review 无法验证。

**循环机制**

```
kf-code-review-graph 启动
  |
  └─ Loop (max_rounds=3):
       ├─ 1. Review → issue_list.md
       ├─ 2. 判断退出：
       │   ├─ issue_count == 0 → 通过
       │   ├─ round >= max_rounds → 标记未解决 issue
       │   └─ 否则继续
       ├─ 3. Agent 修复所有 issue
       └─ 4. 回到 1
```

**退出条件**：
- issue_count == 0 → 通过
- round >= 3 且递减 → 通过，附加 known_issues.md
- round >= 3 且持平/上升 → 中断，标记 flag
- 任何一轮引入 P0 → 人类介入

**集成方式**：Stage 4 入口不变，skill 内部实现 loop，对外部 Pipeline 透明。

### 3. Spec 结构化增强

**立场：大幅增强 Spec 模板，增加验收条件、边界情况、测试策略、性能约束、安全考量、依赖关系、回滚计划。**

**增强模板字段**：

| 字段 | P0功能 | P1功能 | P2功能 | bug修复 |
|------|--------|--------|--------|---------|
| 验收条件(AC) | 强制 | 强制 | 强制 | 强制 |
| 边界情况(EC) | 强制 | 可选 | 可选 | — |
| 测试策略 | 强制 | 强制 | 可选 | 仅回归 |
| 性能约束 | 可选 | — | — | — |
| 安全考量 | 强制 | 可选 | — | — |
| 依赖关系 | 强制 | 可选 | 可选 | — |
| 回滚计划 | 强制 | 可选 | — | — |

---

### 4. 测试前移

**立场：测试贯穿始终，从 Stage 0 定义策略到 Stage 5 冒烟测试。**

**分阶段方案**：
- Stage 0: 在 Spec 中定义 AC + Test Strategy
- Stage 1: 单元测试随代码生成 → 自动执行 → 通过才提交 (>80% coverage)
- Stage 2: 集成测试自动执行 → Mock 外部依赖 → 验证交互契约
- Stage 3: E2E (保持 browser-ops) → 基于 AC 生成用例
- Stage 4: 回归测试 + review-fix 联动 → 每次 fix 后执行受影响模块测试
- Stage 5: 部署后冒烟测试

---

### 5. 集成方案

Plan 审批关卡应插入在 **Stage 0 之后、Stage 1 之前**：

```
Pre-Stage: kf-prd-generator
Stage 0:   kf-alignment + kf-spec → spec.md + plan.md
Gate:      PLAN_APPROVAL → approved-plan.md
Stage 1-5: 三队并发（基于 approved-plan.md）
```

推荐方案 A：修改 `kf-multi-team-compete` Pipeline 编排插入 gate node。
方案 B：新增 `kf-plan-review` 技能（职责单一但增加调用链复杂度）。

### 6. 优先级 Roadmap

| 优先级 | 优化项 | 理由 | 工作量 |
|--------|--------|------|--------|
| **P0** | Plan 审批关卡 | 稳的第一道闸门 | 2-3天 |
| **P0** | Spec 增加 AC + Test Strategy | 与审批联动，可执行 Spec 的前提 | 1天 |
| **P1** | Review-Fix 反馈环 | 显著提升代码质量 | 2-3天 |
| **P1** | 单元测试随代码生成 | 性价比最高的测试前移 | 1-2天 |
| **P2** | 集成测试自动执行 | 依赖单元测试就绪 | 2-3天 |
| **P2** | 回归测试与 review 联动 | 依赖 review-fix 落地 | 1-2天 |
| **P2** | 智能默认批准 | 需要审批历史数据 | 3-5天 |
| **P2** | Stage 5 冒烟测试 | 依赖 CI/CD | 1-2天 |

### 风险提示

1. **效率下降**：审批每次约 1-5 分钟，高频小改动可能不合算。缓解：`--auto` 标志。
2. **审批疲劳**：每个任务都审批可能让用户走过场。缓解：聚焦 scope 和 AC，不纠缠细节。
3. **Spec 过度设计**：写文档代替写代码。缓解：P0 只强制 AC + Test Strategy，其余可选。
4. **测试维护成本**：改代码时要同步更新测试。缓解：review-fix 纳入测试一致性检查。
5. **审批成为瓶颈**：串行审批阻塞后续 Agent。缓解：不同模块的 Plan 可独立审批。
6. **习惯冲突**：用户已习惯"一条命令做到完"。缓解：默认开启但可关闭，用数据证明价值。
