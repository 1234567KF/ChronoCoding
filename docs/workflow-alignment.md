# 最佳实践工作流 — 对齐文档（V3）

> 核心理念：**入口只有一个 `/夯`**，夯自动判断任务类型、自动编排流程、自动引导用户选择深度。
> 人类只在关键决策点做选择确认，不手动触发中间步骤。
> **V3 新增**：深度选择、测试专家循环、进展看板、中断恢复。

---

## 1. 总体流程：从一句话需求到交付

```
用户输入: /夯 [任何需求]
  │
  ├─ 夯 判断任务类型
  │   ├─ bugfix / 小改动 → 直接开发（跳过所有流程）
  │   └─ 系统/模块/新功能 → 进入分步流程
  │
  ├─ [自动] Step 0: 深度选择  ← V3 新增
  │   ├─ Team Lead 展示三档深度:
  │   │   A. 需求 + 方案评审（产出：PRD + 方案对比，不编码）
  │   │   B. 需求 + 设计     （产出：PRD + 架构方案，不编码）
  │   │   C. 全流程编码交付   （产出：代码 + 测试 + 审查）
  │   ├─ 状态持久化到 hang-state.json
  │   └─ 选 A/B 则进入简化流程，选 C 进入完整 Pipeline
  │
  ├── [A 路径] Step 1 方案评审 → 结束
  │   └─ 后续可任何时间恢复:
  │       ├─ 自然对话继续
  │       ├─ gspowers 引导
  │       └─ 夯启动编码 Pipeline
  │
  ├── [B 路径] Step 1+2 → 架构设计 → 结束
  │   └─ 同上恢复选项
  │
  └── [C 路径] 完整 Pipeline:
       │
       ├── [自动] Step 1: 方案评审
       │   ├─ 三队并发出方案 → 裁判评分 → 排名
       │   └─ 人类: 看方案对比 → 选方向 → 确认
       │
       ├── [按需] Step 2: PRD 生成
       │   ├─ kf-prd-generator（需求足够模糊时自动触发）
       │   ├─ kf-alignment 产前对齐
       │   └─ 人类: 确认 PRD / 跳过
       │
       ├── [按需] Step 3: UI 原型（可选）
       │   ├─ kf-ui-prototype-generator
       │   └─ 人类: 看原型提意见
       │
       ├── [自动] Step 4: Spec 生成 + 审查闭环
       │   ├─ kf-spec 生成 spec.md
       │   ├─ spec-reviewer agent 审查 Spec
       │   │   ├─ 检查: AC 完整性、边界覆盖、可测试性、依赖完整性
       │   │   └─ 不达标 → 修复 → 重新审查 → 循环直到达标
       │   ├─ 达标后: 人类最终确认
       │   └─ 人类角色: 只看结果，不参与 AI 审查循环
       │
       ├── [自动] Step 5: 三队开发执行
       │   ├─ 基于 approved spec.md
       │   ├─ 浅层 Plan 注入 → agent prompt（你可 30s 内打断）
       │   ├─ Stage 2 编码（三队并行）
       │   ├─ Stage 3 测试专家循环 ← V3 核心改进
       │   │   ├─ 多角色/多权限/多数据状态测试数据准备
       │   │   ├─ 测试 → issue_list → fix → 回归（上限 3 轮）
       │   │   └─ UI 视觉检查（截图对比 + 布局断言）
       │   ├─ Stage 4 review 条件重审
       │   └─ 汇总交付
       │
       └── [贯穿] 进展看板 ← V3 新增
           ├─ Team Lead 每次交互展示 Pipeline 状态
           ├─ 各队进度百分比 + 阶段产物路径
           └─ fast 跳过 / status 刷新 / stop 暂停
```

## 2. 步骤详解

### Step 0 — 深度选择（V3 新增）

| 项目 | 内容 |
|------|------|
| **触发条件** | 夯判断需要分步流程时 **自动展示** |
| **执行** | Team Lead 展示三档选择，用户回复 A/B/C |
| **状态持久化** | 写入 `.claude-flow/hang-state.json`（深度、当前阶段、已完成阶段、产物路径） |
| **恢复机制** | 后续 `/夯` 检测到已有状态 → 展示三档恢复选项：继续对话 / gspowers 引导 / 夯编码 |
| **默认值** | 10s 无响应 → 默认选 C（全流程编码） |

**深度选择看板**：

```
┌─ 夯 执行深度选择 ──────────────────────────┐
│                                              │
│  你想做到哪一步？                             │
│                                              │
│    A. 需求分析 + 方案评审                     │
│       产出: PRD + 方案对比报告                │
│       后续: 可随时恢复继续                    │
│                                              │
│    B. 需求 + 设计                             │
│       产出: PRD + 架构方案 + 数据模型         │
│       后续: 可随时恢复继续                    │
│                                              │
│    C. 全流程编码交付（默认）                   │
│       产出: 代码 + 测试报告 + 审查报告        │
│                                              │
│  请回复 A/B/C（默认 C）                      │
└──────────────────────────────────────────────┘
```

### Step 1 — 方案评审（自动）

| 项目 | 内容 |
|------|------|
| **触发条件** | 任务明确是系统/模块/新功能时 **自动执行** |
| **执行** | 夯三队并发（方案评审模式）：需求对齐 → 调研 → 方案输出 → 裁判评分 |
| **人类角色** | 看方案对比排名 → 选方向 / 博采众长 |
| **产出** | `方案评审报告.md` |
| **跳过条件** | 用户指定 `--fast`、或需求极其明确简单 |

### Step 2 — PRD 生成（按需自动）

| 项目 | 内容 |
|------|------|
| **触发条件** | 夯判断需求模糊/复杂/跨功能时 **自动触发** |
| **执行** | `kf-prd-generator` → `kf-alignment` 对齐 |
| **人类角色** | 确认 PRD 内容 → 修改意见 → 批准。也可跳过 |
| **产出** | `PRD.md` |
| **跳过条件** | 需求已足够清晰（用户直接给了 Spec 或详细描述） |

### Step 3 — UI 原型（按需可选）

| 项目 | 内容 |
|------|------|
| **触发条件** | 夯判断涉及 UI 变更时 **询问**是否需要原型 |
| **执行** | `kf-ui-prototype-generator` 或 `kf-image-editor` |
| **人类角色** | 看原型 → 确认/修改 |
| **产出** | HTML 原型 / 截图 |
| **跳过条件** | 纯后端 / 用户说不需要 |

### Step 4 — Spec 生成 + AI 审查闭环

```
夯 检测到需要开发编码
  │
  ├─ kf-spec 生成 spec.md（基于 PRD/方案/需求）
  │   └─ 内含: 功能列表、AC、边界情况、测试策略、数据模型
  │
  ├─ spec-reviewer agent（模型: pro）
  │   ├─ 审查维度: AC 二值可测、边界覆盖、测试策略可执行、依赖完整
  │   ├─ 评分: Spec 质量评分卡（自定义标准）
  │   └─ 达标标准: 评分 >= 70, 无 P0 缺陷
  │
  ├─ 不达标 →
  │   ├─ reviewer 出 issue_list
  │   ├─ kf-spec 修复 → 重新生成
  │   └─ reviewer 重新审查 → 循环直到达标（上限 3 轮）
  │
  ├─ 达标后 →
  │   └─ 人类: 展示 Spec + 质量评分 → "请确认"
  │       ├─ approve → 锁定 spec.md，进入 Step 5
  │       ├─ edit → 人类提修改意见 → 修复后重新提交
  │       └─ reject → 返回重新评估需求
  │
  └─ 人类角色: 最终把关者，不参与 AI 审查循环
```

| 项目 | 内容 |
|------|------|
| **触发条件** | 夯判断需要编码开发时 **自动执行** |
| **自动调用** | `kf-spec`（生成）+ `spec-reviewer` agent（审查），**并行** |
| **审查模型** | spec-reviewer 使用 pro 模型（review 需要更强的推理），spec 生成用 flash（效率优先） |
| **循环上限** | 3 轮，超限则升级为人类介入 |
| **人类角色** | Spec 通过 AI 审查后 → 做最终确认（Y 式审核，非 X 式审核） |
| **产出** | `spec.md` + `spec-quality-report.md`（评分 + 审查记录） |

### Step 5 — 三队开发执行（测试专家循环为核心改进）

```
Stage 2 编码完成
  │
  └─ Stage 3: 测试专家循环 ← V3 核心改进
       │
       ├─ Round 1: 测试数据准备 + 首次测试
       │   ├─ 角色数据: 管理员、普通用户、游客
       │   ├─ 权限场景: 有权限/无权限/边界权限
       │   ├─ 数据状态: 空数据、临界值、异常数据、大数据量
       │   ├─ UI 视觉检查: 截屏 + 布局断言（宽屏/窄屏）
       │   └─ 输出: {team}-03-issues.md
       │
       └─ 循环门控:
            ├─ issue_count == 0 → 通过，进入 Stage 4
            ├─ round < 3 → 开发 fix → 回归测试 → 下一轮
            ├─ round >= 3 且递减 → 通过 + known_issues.md
            └─ round >= 3 且持平 → UNRESOLVED + escalation
```

| 项目 | 内容 |
|------|------|
| **触发条件** | Spec 确认后自动进入 |
| **执行** | 夯三队 Pipeline（编码开发模式） |
| **测试专家** | 每队独立测试专家，专注多场景数据 + 循环验证 |
| **UI 视觉检查** | 截图对比 + 布局断言（宽屏/窄屏响应式），无浏览器环境则跳过 |
| **自动调用链** | `kf-model-router` → `claude-code-pro` → 三队并行 Stage 0~5 |
| **人类角色** | 查看交付结果 → 确认/修改 |

## 4. 三个模式

| 模式 | 触发 | 适用场景 | 深度选择 | Spec 审查 | 测试循环 | 测试伴随 |
|------|------|---------|---------|----------|---------|---------|
| **快速** | `/夯 --fast fix bug` | 修复、小改 | 跳过 | 跳过 | 跳过 | 跳过 |
| **均衡** | `/夯 实现xx`（默认） | 大多数开发 | 展示深度选择 → 默认 C | Spec + 1轮审查 | P0 触发重审 + 1 轮测试循环 | 关闭 |
| **增强** | `/夯 --thorough 实现支付` | 核心模块 | 展示深度选择 → 锁定 C | Spec + 3轮审查 | 完整测试循环（3 轮） | 开启 |

## 5. 对比：当前用法 vs 终局形态（V3）

| 维度 | 当前用法 | 终局形态（V3） |
|------|---------|---------------|
| **入口** | 多个命令 | **一个 `/夯`**，自动判断 + 深度选择 |
| **深度选择** | 无 | 展示三档（A 需求/B 设计/C 编码），持久化可恢复 |
| **Spec 生成** | 用户手动触发 | 夯自动检测 → 自动生成 |
| **Spec 审查** | 用户自己看、自己审 | AI reviewer 并行审查 + 循环达标 → 给人最终确认 |
| **测试** | Stage 3 一次性集成测试 | 测试专家循环: 多角色数据 + 多轮测试闭环 + UI 视觉检查 |
| **进展可见性** | 无 | 实时看板: 阶段状态 + 各队进度 + 产物路径 |
| **中断恢复** | 无（重来） | 状态持久化 + 三档恢复选项（对话/gspowers/夯） |
| **用户角色** | 手动每一步 | **只在关键决策点做选择题** |

## 7. 一句话总结

**入口只有一个 `/夯`。夯先问你想做到哪一步（A 需求/B 设计/C 全流程），然后自动引导你走完。选 A/B 可以随时恢复——继续对话、用 gspowers 导航、或用夯启动编码。全流程模式下，测试专家帮你用真实角色数据反复测，测试报告不再骗人。每一步都有看板告诉你现在在哪、还剩什么。**
