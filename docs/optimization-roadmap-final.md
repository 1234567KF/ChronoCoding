# ListenHub 启发下的 kf- 技能体系优化终版方案

> 生成方式：夯 多团队竞争评审 | 日期：2026-05-10
> 裁判评分：蓝队 🥇 8.25 > 绿队 🥈 7.85 > 红队 🥉 6.55
> 对抗质疑：8 维度审查，4 条 MUST-FIX 全部处理

---

## 对抗者质疑处理记录

| MUST-FIX | 决策 | 改动/理由 |
|----------|------|----------|
| **1. 非 kf-spec 技能被完全忽略** | **采纳** | 方案确实过度聚焦 kf-spec，忽略了 20+ 自建技能和 66 上游技能。**终版方案改为"技能无关的轻量通用层"而非"改造每个技能"**。 |
| **2. 查阅式 Plan 无消费端** | **部分采纳** | 对抗者说得对——Plan 没消费者就是垃圾数据。**改为"按需 Plan"模式**：只在 `/夯` 多 Agent 编排时生成聚合 Plan（有消费端），单个技能 / 单 Agent 场景不生成 Plan。Plan 写入 hammer-artifacts 而非独立目录。 |
| **3. Phase 2 风险评分模型是预制债务** | **采纳** | 数据基础设施确实不存在。**风险评分模型移到 P2 末尾 / 降级为 P3**。先建立统一的技能产出 Schema（`{artifact, quality_signals, severity}`），积累 4-6 周数据后再上模型。 |
| **4. 条件 review P0/P1 无实际定义** | **采纳** | 当前 kf-code-review-graph 输出的 severity 未标准化。**Phase 0 先标准化 review 产出格式**，然后条件触发才能工作。反对抗者一并采纳：review 的 severity 字段 MUST 是机器可读的枚举值。 |

### 其他对抗意见处理

| 质疑 | 决策 | 理由 |
|------|------|------|
| **缓存失效问题**（Plan 差异化 → 缓存命中率骤降） | **采纳** | DeepSeek 缓存 120x 价差（Pro ¥3/M vs ¥0.025/M），这是经济问题不是技术问题。**终版方案中所有 Plan/质量信号输出 MUST 追加在共享前缀之后、差异化分节中**，保持前缀一致性。 |
| **66 个上游技能不受控** | **采纳** | 上游技能跳过所有优化。**终版方案明确优化范围只覆盖 kf- 自建技能**，上游技能只做兼容性不破坏。 |
| **用户认知负担被低估** | **部分采纳** | 用户确实不需要学 AC/Edge Cases/风险评分等概念。**终版方案用"一条命令 + 自动适配"原则**：新增 `--thorough` flag，默认 `--balanced`（只启用 P0 优化），用户只需记住一个 flag。 |
| **风险评分过度工程** | **采纳** | 初始阶段不用评分模型。**替代方案：简单规则引擎**——文件数量 + 安全关键词匹配 → 三档分类。够用且零基建。 |

## 终版方案

### 核心理念

**不是改造每个技能，而是在技能之间建立一个轻量质量协调层。**

当前 kf- 体系的真正瓶颈不是"某个技能不够强"，而是**技能之间缺乏统一的 Plan 路径、Review 标准和质量信号**。优化目标不是让 kf-spec 变得更强，而是让从"需求输入"到"代码交付"的全链路有 visibility、有反馈、有兜底。

### 优化范围

| 覆盖 | 范围 |
|------|------|
| ✅ 覆盖 | 所有 kf- 自建技能（20+）——通过统一质量协调层，非逐个改造 |
| ✅ 覆盖 | `/夯` 多 Agent 编排场景——聚合 Plan + 聚合 Review |
| ❌ 不覆盖 | 66 个上游 jeffallan 技能——保持兼容，不做改造 |
| ❌ 不覆盖 | 非结构化 CLI 场景——用户直接对话时无额外流程 |

## 分阶段 Roadmap

### Phase 0 — 基础设施（Week 1, P0）

**目标**：解决对抗者提出的"无消费端、无数据、无标准"三个先决问题。

| 任务 | 说明 | 工作量 |
|------|------|--------|
| **0.1 统一技能产出 Schema** | 所有 kf- 技能输出的 artifact 追加 `quality_signals` 元数据块：`{artifact_type, severity, changed_files[], test_status}`。不影响功能，只追加元数据。 | 2-3 天 |
| **0.2 Review severity 标准化** | kf-code-review-graph 输出的问题列表改为机器可读的枚举：`P0/P1/P2/P3`。当前是自由文本 → 改为结构化 Markdown 表格 + JSON 双格式。 | 1 天 |
| **0.3 浅层 Plan 路径** | 不生成 Plan 文档。改为 `/夯` spawn agent 时在 prompt 前 10 行注入"任务拆解预览"（结构化任务列表），人类可打断修改。**不是文档，是 prompt 的一部分**。 | 1 天 |
| **0.4 缓存审计** | 检查所有 agent prompt 的前 300 token 一致性，确保 Phase 0 改动不破坏 DeepSeek 缓存前缀。 | 0.5 天 |

**Phase 0 门控**：统一 Schema 和 severity 标准化完成后才进入 Phase 1。

---

### Phase 1 — 质量闭环（Week 2, P0）

**目标**：以最小 token 增量实现"有反馈、有兜底"。

| 任务 | 说明 | 工作量 |
|------|------|--------|
| **1.1 Spec AC 字段实验** | kf-spec 以 feature flag 方式增加 AC（验收条件）字段，默认关闭。用户可通过 `/spec --ac` 启用。运行 2 周看采用率。 | 1 天 |
| **1.2 条件 review 重审** | 基于 Phase 0.2 的标准化 severity：review 中出现 P0 → 自动触发 1 轮重审；P1 密度 > 3/KLOC → 触发重审；否则一次过。上限 3 轮。 | 1-2 天 |
| **1.3 `/夯` 浅层 Plan** | `/夯` 启动时在 agent prompt 注入结构化的任务拆解预览（10-15 行），人类可打断修改。不写 Plan 文件。 | 1 天 |
| **1.4 单元测试伴随** | 编码 agent 收到 `--with-tests` flag 时：每生成一个函数/组件，同步生成单元测试骨架，执行通过后再提交。默认关闭。 | 1-2 天 |

**Phase 1 门控**：AC 实验 2 周后评估采用率 > 30% 则默认打开，否则维持 feature flag。

---

### Phase 2 — 质量信号（Week 3-4, P1）

**目标**：建立数据基线，但不建平台。

| 任务 | 说明 | 工作量 |
|------|------|--------|
| **2.1 质量信号聚合** | 每个任务完成后，自动汇总该任务的 review 评分、测试通过率、问题密度。输出到 `.claude/logs/quality-signals.jsonl`（追加日志，零基建）。 | 1 天 |
| **2.2 简单风控规则** | 不建评分模型，用规则引擎：变更文件 > 5 且含 auth/payment 关键词 → 自动提醒人类关注。规则硬编码，写死在 `.claude/quality-rules.json`。 | 1 天 |
| **2.3 缓存经济性检查** | 每次改动后执行 `node .claude/helpers/cache-audit.cjs`（新增脚本），确保共享前缀没被破坏。 | 1 天 |
| **2.4 关键路径集成测试** | 只在"核心流程"（登录→操作→退出）做集成测试前移。规则写在 `pipeline-config.yaml` 中。 | 2 天 |

**Phase 2 门控**：质量信号累积 > 100 条记录后开始校准规则阈值。

---

### Phase 3 — 数据驱动优化（Week 5-8, P2）

**目标**：基于 Phase 2 数据做精准优化。

| 任务 | 触发条件 | 说明 |
|------|---------|------|
| **3.1 风险评分模型** | Phase 2 信号 > 200 条 | 基于真实数据训练简单线性评分模型（非 LLM），替换 Phase 2 的硬编码规则 |
| **3.2 Plan 审批** | 模型校准后，误拦率 < 10% | 仅在评分模型判定 R3 时启用 Human-in-the-loop 审批 |
| **3.3 全量测试前移** | 单元测试覆盖率 > 80% 维持 1 个月 | 把集成测试从 Stage 3 前移到 Stage 2 |
| **3.4 质量仪表盘** | 信号 > 500 条 | 简单的 HTML 报表，从 jsonl 数据渲染 |

**Phase 3 门控**：没有数据就没有模型，没有覆盖就没有前移。

## 关键决策原则

| 原则 | 说明 |
|------|------|
| **技能无关** | 优化方案不假设 kf-spec 的存在。所有改动基于通用质量协调层，对技能类型无感。 |
| **有消费端才生产** | 不生产任何没人读的数据。Plan/质量信号/评分都必须有明确的消费端（Agent/人类/Pipeline）。 |
| **实验先行** | 每个新功能以 feature flag 方式上线，运行 2 周评估效果后决定是否全量。 |
| **不破坏缓存** | 所有优化必须保持 agent prompt 前 300 token 不变，保护 DeepSeek KV Cache 经济性。 |
| **用户无感（默认时）** | 默认模式下用户不应感知到任何新流程。所有新功能通过 `--thorough` / `--with-tests` 等 flag 控制。 |

## 碾压指标

| 维度 | 当前状态 | Phase 1 后 | Phase 3 后 |
|------|---------|-----------|-----------|
| 方向性返工 | 当前无拦截 | P0 review 拦截 60-80% | + 高风险 Plan 审批 |
| Review 闭环率 | 0%（一次过） | P0/P1 重审覆盖率 100% | 自适应闭环 |
| Spec 可执行性 | 偏概要 | AC 字段实验（30%+ 采用率为目标） | 数据驱动优化 |
| 测试覆盖 | Stage 3 E2E 仅 | + 单元测试伴随（实验） | 全量集成前移 |
| Token 增量 | 基线 | ~5-8%（检索+判断开销） | ~10-15%（含质量信号） |
| 用户认知负担 | 最小 | 1 个新 flag（`--thorough`） | 同左 |

## 执行顺序速查

```bash
# 日常开发（默认，零变化）
/夯 实现登录功能

# 质量增强模式（新增，可选）
/夯 --thorough 实现支付模块

# 快速模式（跳过所有检查）
/夯 --fast 修复 typo
```

**一句话总结**：不是改造每个技能，而是在技能之间建立轻量质量协调层。有消费端才生产，有数据才决策，不破坏缓存，不增加用户认知负担。
